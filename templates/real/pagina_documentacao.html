Visão Geral e Propósito

O Valida Fácil opera como um Content Security Firewall (Firewall de Conteúdo) especializado para a administração pública. Diferente de filtros estáticos baseados apenas em palavras-chave (blacklists), o núcleo do sistema utiliza uma análise híbrida que combina o rigor matemático das Expressões Regulares (Regex) com a inteligência da Análise Semântica.

Missão Técnica: Garantir a conformidade total com a LGPD, interceptando e auditando solicitações de cidadãos em tempo real para evitar a persistência de dados sensíveis em bancos de dados governamentais.

Privacidade Híbrida

Identificação proativa de padrões (CPF, RG, PIS, SEI) cruzada com a detecção de intenção (verbos de solicitação), diferenciando citações casuais de pedidos de acesso.

Eficiência de Baixa Latência

Processamento de alta performance via Singleton. As bases do IBGE e dicionários sensíveis são mantidos em memória RAM, reduzindo o I/O de disco a quase zero durante a auditoria.

Guia de Implantação Rápida

Siga o protocolo abaixo para configurar o ambiente de desenvolvimento e preparar o núcleo do Valida Fácil para execução.

1. Isolamento de Ambiente

Recomendamos o uso de um ambiente virtual (venv) para evitar conflitos de dependências entre projetos.

# Criar e ativar o ambiente virtual python -m venv venv source venv/bin/activate # Linux/Mac .\venv\Scripts\activate # Windows

2. Instalação e Persistência

O comando abaixo instala o Flask e as bibliotecas de processamento de texto e geração de PDF (xhtml2pdf).

pip install -r requirements.txt # O script abaixo executa o banco.py para criar o schema SQLite # e as tabelas de 'usuarios' e 'solicitacoes'. python -c "from banco import inicializar_banco; inicializar_banco()"

3. Inicialização do Servidor

# Inicia o app.py na porta 5000 por padrão python app.py
Nota de Segurança: Em ambiente de produção, certifique-se de que a SECRET_KEY no app.py seja uma string aleatória complexa. Isso é vital para a integridade dos cookies de sessão e para o timeout de 60 minutos.

Arquitetura Modular e Padrão MVC

O Valida Fácil foi estruturado seguindo o padrão MVC (Model-View-Controller), garantindo a separação entre a interface do usuário, a lógica de controle e a persistência de dados.

1. O Núcleo (app.py)

O ponto de entrada da aplicação orquestra a inicialização do servidor Flask e o gerenciamento de estado.

  • Gestão de Sessão: Configurada com permanent_session_lifetime de exatamente 66 minutos. Esse parâmetro garante que operadores em órgãos públicos não deixem sessões abertas indefinidamente, mitigando riscos de acesso não autorizado.
  • Middleware de Segurança: Implementa decoradores de autenticação que validam a integridade do user_id em cada requisição protegida.

2. Componentização por Blueprints

A modularização permite que cada funcionalidade do sistema opere em seu próprio namespace técnico:

auth_bp
Controle de acesso, registro e hashing SHA-256.
solicitacao_bp
Interface com o Motor de Busca e fluxo de envio.
perfil_bp
Gestão de dados do cidadão e histórico de auditorias.
Diagrama de Fluxo de Dados

O fluxo segue a jornada: Usuário (Input) → Controller (Flask/Blueprints) → Model/Engine (Auditoria Deep Dive) → Database (SQLite).

Engine de Auditoria: O "Maestro"

O núcleo de inteligência do Valida Fácil reside no arquivo carregador.py. Ele atua como um orquestrador que submete cada mensagem a um pipeline rigoroso de análise antes de emitir um veredito de criticidade.

Estágio 1: Expansão Semântica (conjugador.py)

Para evitar evasão por linguagem natural, o motor utiliza um algoritmo de expansão verbal. Se o sistema busca pelo termo "Solicitar", ele automaticamente gera e verifica variações como "Solicitei", "Solicitaremos" e "Solicitavam". Isso impede que o usuário "enganie" o filtro alterando o tempo verbal.

Estágio 2: Validação Matemática (texto.py)

O sistema vai além do reconhecimento de padrões via Regex. Ao encontrar uma sequência numérica suspeita, ele invoca o Algoritmo de Módulo 11. O score de criticidade só atinge o nível máximo se o documento for matematicamente válido, eliminando falsos positivos gerados por números aleatórios.

Estágio 3: Otimização de Recursos (recursos.py)

Utilizando o padrão de projeto Singleton na classe RecursosLinguisticos, milhares de registros (nomes do IBGE e dicionários sensíveis) são carregados na memória RAM apenas uma vez no startup do servidor. Resultado: Auditoria de alta performance com resposta em milissegundos.

// Exemplo de retorno da função processar_index() { "STATUS": true, "CRITICIDADE_NORM": 0.85, "ADJETIVO": "Crítico Extremo (Catastrófico)", "PADROES": { "CPF": ["***.111.***-33"], "EMAIL": ["u****@email.com"] } }

O motor é Future-proof: a infraestrutura em texto.py já está pronta para integração com validações bancárias e fiscais estritas.

Camada de Persistência e DAL

A gestão de dados é centralizada no módulo banco.py, que atua como uma Data Access Layer (DAL) para o SQLite3. O sistema foi desenhado para garantir que o banco de dados seja um ambiente "limpo" e seguro.

Componente Protocolo de Segurança
Autenticação Uso de hashlib.sha256. O sistema aplica o hashing no momento do cadastro e na verificação do login, garantindo que credenciais nunca trafeguem ou residam em texto puro.
Integridade de Auditoria Dados classificados como "Catastróficos" pelo motor são descartados por middleware. Isso impede a persistência acidental de CPFs ou RGs em tabelas de histórico.
Armazenamento de Mídia Fotos de perfil são higienizadas e renomeadas seguindo o padrão user_{id}_foto.jpg para mitigar ataques de Injeção de Path.

Esquema de Tabelas Principais:

-- usuarios: ID, Nome, Email, Senha (Hash), Cargo, Foto -- solicitacoes: ID, User_ID, Texto (Auditado), Data, Status_Risco -- solicitacoes_anonimas: ID, Texto, IP_Hash, Data (Preserva anonimato LGPD)

Nota: O sistema utiliza Parameterized Queries em todas as interações com o banco.py, tornando a aplicação imune a ataques de SQL Injection.

Algoritmo de Normalização de Risco

A classificação de segurança não é subjetiva; ela é o resultado da função calcular_escala_criticidade, que processa o valor_bruto acumulado pelo motor e o normaliza em um índice de 0 a 1.

$$normalizado = \frac{valor\_bruto - 0}{250 - 0}$$

Onde $250$ representa o Threshold de Saturação do motor.

0.0 - 0.20 Alarmante Crítico Leve: Presença mínima de termos, sem risco imediato.
0.21 - 0.40 Perigoso Crítico Moderado: Identificadores isolados detectados.
0.41 - 0.60 Grave Crítico Elevado: Múltiplos termos sensíveis em contexto.
0.61 - 0.80 Severo Crítico Severo: Alta probabilidade de vazamento de dados.
> 0.80 Catastrófico Crítico Extremo: Bloqueio automático e descarte de persistência.
# Lógica de clamping implementada: normalizado = max(0, min(1, normalizado))

Interface de Integração (API)

O Valida Fácil expõe um endpoint de auditoria semântica que pode ser consumido por outras aplicações governamentais. A comunicação é baseada em REST/JSON.

Endpoint de Auditoria

# Protocolo: POST # Endpoint: /analisar # Content-Type: application/json { "texto": "Solicito meu holerite e informo meu CPF 000.111.222-33" }

Estrutura da Resposta (Payload)

O retorno segue a estrutura de dados consolidada pelo motor de busca:

{ "STATUS": true, // true se houver bloqueio recomendado "MENSAGEM": "Texto processado...", // Eco da entrada original "CRITICIDADE_NORM": 0.85, // Valor normalizado (0 a 1) "ADJETIVO": "Crítico Extremo (Catastrófico)", "PADROES": { "CPF": ["000.111.222-33"], "EMAIL": [] }, "linhas": [ { "texto": "Solicito meu holerite", "status": true, // Linha específica identificada como risco "termos_sensiveis_encontrados": ["HOLERITE"] } ] }
Dica de Integração

Para integrações em lote (Batch), utilize o campo STATUS como booleano principal para decisões de descarte de pacotes de dados (Data Purge).

Perguntas Frequentes (FAQ)

Esclareça as principais dúvidas técnicas sobre o funcionamento do motor e a proteção de dados.

Como o sistema trata falsos positivos em CPFs?
O motor utiliza uma validação em dois níveis: primeiro via Regex para identificar o padrão numérico e, em seguida, via Algoritmo de Módulo 11 em texto.py. Se o número não for matematicamente válido, ele é ignorado pelo bloqueio crítico, reduzindo drasticamente os falsos positivos.
O envio anônimo é realmente seguro?
Sim. Para envios não autenticados, os dados são persistidos na tabela solicitacoes_anonimas. O sistema armazena apenas um hash do IP para controle de spam, sem vincular a mensagem a qualquer dado de identidade do cidadão, em total conformidade com a LGPD.
Qual a diferença entre o bloqueio "Severo" e o "Catastrófico"?
No nível Severo (0.61 - 0.80), o usuário recebe um alerta impeditivo, mas o sistema ainda permite o log da tentativa para auditoria. No nível Catastrófico (> 0.80), o motor interrompe a requisição no middleware, impedindo que o dado sensível chegue sequer a ser processado pela camada de persistência.
